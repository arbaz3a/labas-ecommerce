\section{Authentication}
This API uses \textbf{JSON Web Tokens (JWT)} for authentication.
\begin{itemize}
    \item \textbf{Obtaining a token:} Tokens are returned upon successful login (\texttt{/api/auth/login} or \texttt{/api/admin/login}) or registration (\texttt{/api/auth/register}, \texttt{/api/admin/setup}). The token is typically provided in a cookie and/or JSON response depending on the \texttt{sendTokenResponse} utility.
    \item \textbf{Sending the token:} Protected routes expect the token in the \texttt{Authorization} header as a Bearer token: 
    \texttt{Authorization: Bearer <your\_jwt\_token\_here>}
    \item \textbf{Invalid/Expired Tokens:} Requests with missing, invalid, or expired tokens will receive a \texttt{401 Unauthorized} response. Accessing admin routes with a regular user token will return a \texttt{403 Forbidden} response.
\end{itemize}

\section{File Uploads}
\begin{itemize}
    \item Endpoints handling file uploads expect a \texttt{multipart/form-data} request.
    \item \textbf{Product Images:} \texttt{POST /api/admin/upload/product} accepts an array of files under the field name \texttt{images} (maximum 10 files).
    \item \textbf{Allowed Types:} JPG, PNG, WebP.
    \item \textbf{Size Limit:} 5MB per file.
\end{itemize}

\section{Notes on Review}
\begin{enumerate}
    \item \textbf{Unused or Inconsistent Routes:}
    \begin{itemize}
        \item The \texttt{/api/admin/settings} endpoint looks for a file \texttt{data/settings.json}, but there is no corresponding \texttt{PUT /api/admin/settings} to ever update this file. It is currently read-only.
        \item The \texttt{/api/auth/me} endpoint returns \texttt{\{ success: true, user \}} but uses \texttt{protect} middleware which sets \texttt{req.user}. It is slightly redundant if the frontend just decodes the JWT, but it's a standard pattern.
        \item For \texttt{/api/admin/orders/:id/status} and \texttt{/api/admin/orders/:id/payment}, there are no \texttt{express-validator} rules defined in \texttt{adminRoutes.js} checking that the status/paymentStatus are valid enums. This relies strictly on Mongoose's \texttt{runValidators: true} inside the controller.
        \item For \texttt{/api/orders} (Create Order), there is no express validation middleware in \texttt{orderRoutes.js} for the complex nested object (items, shippingAddress, paymentMethod); it relies solely on the controller logic.
    \end{itemize}
\end{enumerate}
